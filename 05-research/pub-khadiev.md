## Профиль
- Масштаб: маленькая студия, 2 человека
- Платформы: Android + iOS
- Текущий стек аналитики: CAS кабинет (ежедневно) + Google Analytics (по версиям) + Tenjin (от паблишинга)
- Опыт с CAS: активный паблишер, работает с командой Влада (текущие менеджеры: Рашид, Ирина)
- Роль: сам делает геймдизайн, продюсирование, иногда рекламные креативы. Программист не заходит в кабинет

## Ежедневный ритуал в кабинете

1. **Applications** — проверить, не отлетела ли сеть (policy compliance)
2. **Payments** — проверить корректировки (вычли/добавили)
3. **Publishing** — spend, revenue, разница по всем играм за предыдущий день
4. **Performance Marketing** — основные игры, не упала ли реклама/сетка

**Главный use case:** диагностика — "не отвалилось ли что-то?"

## Ключевые боли

1. **Сети отваливаются без уведомления** — сеть может перестать работать, 0 impressions, и никто не скажет. Он вручную копирует таблицу сетей в свою таблицу и сравнивает. _"Иногда могут отлетать сетки... нужно всегда за ними следить... по каждому приложению проходиться."_ Хочет видеть ВСЕ сети (включая неработающие и неподключённые) в одном виде.

2. **Нет разбивки по версии приложения** — хочет сравнивать метрики между версиями. Сейчас использует Google Analytics, но там только revenue по версиям, а хочет ARPDAU и всё остальное. _"Раньше я думал, что было бы круто добавить, чтобы можно было по версиям сравнивать... в гугл аналитике только доход, а тут и доход на игрока."_

3. **Баг: дата-фильтр слетает** — при переключении периода (Last 7 days → 30 days) фильтр сбрасывается обратно, нужно кликать дважды. _"Ставишь, он потом почему-то обратно на то, которое было ставить, и тебе нужно второй раз это делать."_

4. **Ложные алерты по удалённым приложениям** — удалённые из стора приложения показывают крестик в диагностике Applications, из-за чего в главном меню висит warning badge, хотя все рабочие приложения в порядке.

5. **Нет прозрачности по креативам и UA** — хочет видеть, какие креативы в каких сетях работают. Сейчас спрашивает команду. _"Было бы вообще круто... чтобы я мог зайти в сетку, интеграл, заходишь, мое приложение выбираю... какие рекламные креативы работают."_ Делает креативы сам, хочет учиться на данных.

6. **Скорость реакции команды** — текущая команда (Рашид, Ирина) отвечает медленнее, чем предыдущая. Нет проактивных уведомлений о решении проблемы — приходится самому follow-up делать. _"Раньше чуть ли не в тот день, а сейчас бывает пару дней."_

7. **Раздражающий баннер без кнопки закрытия** — есть UI-элемент, который нельзя скрыть. _"Если бы ещё помните, крестик был, чтобы его выключить."_

## Инсайты для продукта

### Что подтвердилось
- Ежедневная диагностика — главный use case (подтверждает US-L1-01, US-L2-05)
- Разбивка по версиям — нужна даже маленьким студиям (подтверждает US-L2-01..03)
- Аномалии/нули должны подсвечиваться (подтверждает US-L2-21)
- Прозрачность по UA/креативам критична для доверия к паблишеру

### Что нового узнали
- **Network health view** — показывать ВСЕ сети, не только активные. Включая неподключённые (чтобы клиент видел, что ещё можно добавить) и с 0 impressions (чтобы видеть отвалы). Это расширение US-L2-05
- **Cross-app eCPM comparison** — конкретный use case: выбрать страну (USA), тип рекламы (Banner), сравнить eCPM по всем играм. Помогает понять, где приложение недозарабатывает
- **Creative performance self-service** — Pub-разработчики хотят видеть перформанс креативов по сетям. Не для управления UA, а чтобы учиться делать лучшие креативы. _"Если сам делаешь видео, знаешь, как лучше их делать"_
- **Маленькие Pub-студии не используют self-service фичи** (export, saved views, share). Подтверждает гипотезу Жвикова о Pub-lite
- **Доверие к паблишеру** — прозрачность данных = доверие. Если клиент видит все сетки, все креативы, все расходы — он верит, что за него работают

### Что противоречит текущим планам
- Export/Share/Saved Views для маленьких Pub — может быть overengineering. Два из двух Pub-интервью (Жвиков, Хадиев) говорят, что им это не нужно

### BX-инсайты (процесс работы с командой)
- **Скорость реакции деградировала** — текущая команда медленнее предыдущей. Клиент сравнивает
- **Нет проактивных уведомлений** — клиент сам follow-up-ит. Нужен тикет-подобный процесс с автоматическим уведомлением "исправлено"
- **Еженедельный ритм по креативам** — отправляет креативы раз в неделю, ждёт фидбэк. Формализовать?
