# One BI — Словарь терминов

Единый язык для коммуникации между CPO (Alexei), Data (Borys) и Frontend (Ruslan).

## 1. Архитектура: три слоя

```
┌──────────────────────────────────────────────────────┐
│  UI Layer (Ruslan)                                   │
│  Шаблон + виджеты + фильтры + таблицы                │
│  Что видит пользователь                              │
├──────────────────────────────────────────────────────┤
│  Data Layer (Borys)                                  │
│  Materialized Views в ClickHouse                     │
│  Подготовленные таблицы, из которых UI забирает      │

│  Data Sources                                        │
│  Откуда берутся сырые данные                         │
└──────────────────────────────────────────────────────┘
```

## 2. Data Sources — Источники данных

| Термин | Что это | Задержка | Точность | Кто отвечает |
|--------|---------|----------|----------|--------------|
| **API Data** | Данные от рекламных сетей (AdMob, Unity Ads, Meta и т.д.), забираемые по их API | ~1.5 суток | Эталон (подтверждённые сетями) | Борис (сбор) |
| **ILRD** (Impression-Level Revenue Data) | Данные с CAS-сервера: каждый показ рекламы трекается нашим SDK на устройстве и отправляется к нам | Real-time (секунды) | ~95-98% от API (дискреп 1.7-5%) | Борис (сбор + хранение) |

### Ограничения ILRD

| Ограничение | Описание |
|-------------|----------|
| Версия SDK | Только CAS SDK 4.3+ отправляет ILRD. Старые версии — слепая зона |
| Китай | ILRD-трафик из Китая отключён (экономия серверов, монетизация КНР ~0) |
| Дискреп | ILRD обычно **занижает** доход (не все показы долетают). Не завышает |

### Типы данных на таймлайне

```
         API Data (confirmed)              ILRD (expected)
  ◄──────────────────────────────┤──────────────────────►
  ...  21 фев   22 фев   23 фев │ 24 фев   25 фев  сейчас
                                 │
                          последний день    ← "gap" в 1.5 суток
                          с данными от сетей
```

## 3. Data Layer — Что делает Борис

### Materialized Views (MV)

**Что это:** предагрегированные таблицы в ClickHouse, из которых UI забирает данные. Борис берёт сырые данные (API + ILRD), считает метрики и кладёт в готовом виде.

**Зачем:** чтобы UI не считал ничего на лету — просто запрашивает готовую строку из таблицы.

| Термин | Описание |
|--------|----------|
| **MV (Materialized View)** | Готовая таблица с предрассчитанными данными. UI дёргает из неё |
| **View / Вьюшка** | То же, что MV. Борис говорит "я подготовлю вьюшку" = сделаю таблицу с нужными данными |
| **ClickHouse (CH)** | Аналитическая БД, где лежат все MV. Быстрая на чтение |
| **ClickHouse Cloud** | Облачная версия CH, дороже и слабее. Сейчас используется для ILRD |
| **Собственный кластер CH** | Задача Бориса + Стаса: перенести ILRD на свой сервер CH (дешевле) |

### Компенсационные коэффициенты

**Что это:** алгоритм Бориса для сглаживания дискрепа ILRD vs API.

**Как работает:**
1. За закрытые дни (где есть и API, и ILRD) считается коэффициент по каждому разрезу (app + сеть + формат + страна)
2. Этот коэффициент применяется к ILRD-данным за свежие дни
3. Результат: "expected revenue" — прогноз, близкий к будущим API-данным

**Статус:** у Бориса есть рабочий прототип (показывал на маркетинговых дашбордах в Superset).

## 4. UI Layer — Что делает Руслан

### Структурные компоненты

| Термин | Описание | Пример |
|--------|----------|--------|
| **Шаблон** | Купленный UI-темплейт (sidebar, header, layout, компоненты) | Админка с боковым меню |
| **Виджет** | Визуальный блок, отображающий метрику или график | Карточка "Revenue: $48.2K +8.5%" |
| **Таблица** | Табличное отображение данных с сортировкой и пагинацией | Строки по app с колонками Revenue, eCPM, DAU |
| **График** | Chart: line, bar, donut | Линия Revenue по дням |
| **Chip** | Компактный тег для фильтра, сплита или метрики | `[iOS ✕]` `[Country ▼]` |

### Три строки управления

Основной паттерн интерфейса Reports (UX-принцип из design-spec):

```
┌────────────────────────────────────────────────┐
│ Строка 1: Filters   [App ▼] [iOS ✕] [US ✕]    │  ← Что ограничиваем
│ Строка 2: Splits     [Country] [Ad Type]       │  ← Как группируем
│ Строка 3: Metrics    [Revenue] [eCPM] [DAU]    │  ← Что показываем
├────────────────────────────────────────────────┤
│           Виджеты / Таблица / Графики          │
└────────────────────────────────────────────────┘
```

| Строка | Что делает | Откуда берёт список | Документ |
|--------|-----------|-------------------|----------|
| **Filters** | Ограничивает выборку (WHERE) | `04-bi/filters.md` | Filters — Common + Mediation |
| **Splits** | Группировка данных (GROUP BY) | `04-bi/filters.md` | Splits / Group by |
| **Metrics** | Какие числа показывать (SELECT) | `04-bi/metrics-dictionary.md` | По ID метрики |

### Два экрана

| Экран | Для чего | Сложность для Руслана |
|-------|----------|----------------------|
| **Quick View** | "Всё ли ок?" — карточки + тренд + пироги. Минимум настроек | Низкая |
| **Reports** | Полный анализ с тремя строками управления | Высокая |

## 5. Метрики — Что доступно сейчас и что появится

### Фаза 1: API Data (сейчас)

Текущий медиэйшн-дашборд. Все данные от сетей.

**Ключевые метрики:**

| ID | Метрика | Описание |
|----|---------|----------|
| m13 | Ad Revenue | Доход от рекламы |
| m46 | Impressions | Кол-во показов |
| m20 | eCPM | Доход на 1000 показов |
| m5 | DAU | Активные пользователи в день |
| m38 | Ad ARPDAU | Рекл. доход на DAU |
| m21 | Fill Rate | % заполненных запросов |
| m48 | CTR | Кликабельность |

**Ключевые фильтры:** App, Country, OS, Activity Date
**Ключевые сплиты:** Network, Ad Type, Country

### Фаза 2: + ILRD (следующая)

Борис подключает ILRD-данные. Это открывает:

| Что появляется | Почему |
|----------------|--------|
| **Real-time данные** | ILRD приходит в секундах, не ждём сутки |
| **Разбивка по SDK Version** (d1) | ILRD привязан к конкретной версии SDK на устройстве |
| **Разбивка по App Version** (d2) | Какой билд приносит больше/меньше |
| **Active Users** (m37) | Точнее, чем DAU из сетей — считается по нашему трекеру |
| **Session-метрики** (s1-s8) | Длина сессий, сессий на юзера — только из ILRD |

### Фаза 3: + Когортный анализ (будущее)

ILRD привязан к install date, что позволяет:

| Что появляется | Почему |
|----------------|--------|
| **Retention D1-D90** (r1-r13) | Когорты по дате инсталла |
| **LTV** (m12) | ARPDAU x Lifetime Days |
| **Cohort Session Metrics** (cs0-cs8) | Вовлечённость по дням жизни |
| **A/B тесты** (m24-m30) | Сравнение когорт Test vs Control |

## 6. Контракт: как Борис и Руслан общаются

```
Borys                              Ruslan
──────                             ──────
MV с данными  ──── API endpoint ──── UI-компонент
              (REST / SQL view)

Борис отвечает:                    Руслан отвечает:
- Какие колонки в MV               - Как отобразить данные
- Какие фильтры поддерживает       - Chip-ы для фильтров
- Скорость ответа                  - Виджеты / таблицы / графики
- Формат данных                    - Interactivity (сортировка, hover)
```

### Словарь для разговора

| CPO говорит | Борис слышит | Руслан слышит |
|-------------|-------------|--------------|
| "Покажи Revenue по странам" | MV с GROUP BY country, SELECT revenue | Таблица/chart, сплит = Country, метрика = Revenue |
| "Добавь фильтр по сети" | Колонка network в MV, фильтрация по ней | Chip "Network" в строке Filters |
| "Включи real-time" | Подключить ILRD-вьюшки, показывать данные за сегодня | Дополнительные кнопки периода (1h/6h/12h) или пунктирная линия на графике |
| "Нужны сессии" | Session MV из ILRD (только ILRD-источник) | Новые метрики s1-s8 в строке Metrics |
| "Когортный анализ" | MV с install_date, day_since_install | Новый экран/таб с когортной таблицей |
| "Дисклеймер по ILRD" | Метаданные в ответе API: source=api/ilrd | Иконка/тултип "Expected data based on ILRD" |
