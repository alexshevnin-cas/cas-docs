## Схема

```
 Data Source        Metrics Layer           UI
 (откуда)          (что считаем)           (как показываем)
┌──────────┐      ┌───────────────┐      ┌──────────────────┐
│ API Data │─────►│               │      │  Filters (WHERE) │
│ (сети)   │      │   Metrics     │─────►│  Splits (GROUP)  │
│          │      │               │      │  Metrics (SELECT)│
│ ILRD     │─────►│  Borys готовит│      │                  │
│ (CAS)    │      │  Ruslan берёт │      │  Ruslan рисует   │
└──────────┘      └───────────────┘      └──────────────────┘
```

**Metrics Layer** — это контракт. Борис кладёт числа, Руслан забирает и рисует. Не важно, откуда данные — UI одинаковый.

## Фаза 1: API Data → новый UI

Сейчас. Данные от рекламных сетей, задержка ~1.5 суток. Строим новый интерфейс на этих данных.

### Metrics

| Metric | Описание | ID |
|--------|----------|----|
| **Ad Revenue** | Доход от рекламы | m13 |
| **Impressions** | Количество показов | m46 |
| **eCPM** | Revenue / Impressions x 1000 | m20 |
| **DAU** | Уникальные пользователи в день | m5 |
| **Ad ARPDAU** | Ad Revenue / DAU | m38 |
| **Fill Rate** | Fills / Requests | m21 |
| **CTR** | Clicks / Impressions | m48 |
| **Requests** | Запросы рекламы | m49 |
| **Clicks** | Клики по рекламе | m47 |

### Filters (чем ограничиваем выборку)

| Filter | Значения |
|--------|----------|
| **Period** | 7d, 30d, 90d, custom |
| **App** | Одно или несколько приложений |
| **Country** | Страна пользователя |
| **Platform** | iOS / Android |

### Splits (как группируем строки)

| Split | Что делает |
|-------|-----------|
| **Application** | Строка на каждое приложение |
| **Country** | Строка на каждую страну |
| **Network** | Строка на каждую рекл. сеть (AdMob, Unity, Meta...) |
| **Ad Type** | Banner / Interstitial / Rewarded / MREC / Native |
| **Platform** | iOS / Android |

### UI-компоненты

| Компонент | Что это |
|-----------|---------|
| **Quick View** | Карточки с ключевыми числами + тренд-график |
| **Reports** | Таблица + графики с полными фильтрами и сплитами |
| **Metric Card** | Виджет: число + динамика ("+8.5%") |
| **Chart** | График (line / bar / donut) |
| **Table** | Таблица данных с сортировкой |
| **Chip** | Тег-кнопка для фильтра / сплита / метрики |

## Фаза 2: + ILRD → новые Metrics

Следующий шаг. Борис подключает данные CAS-сервера (real-time, задержка секунды). Metrics Layer расширяется.

### Что такое ILRD

Impression-Level Revenue Data — каждый показ рекламы трекается CAS SDK на устройстве и отправляется на наш сервер. Не зависит от API сетей.

| Свойство | Значение |
|----------|----------|
| Задержка | Секунды |
| Точность | ~95-98% от API (занижает, не завышает) |
| Требует | CAS SDK 4.3+ |
| Исключение | Китай (отключён) |

### Новые Metrics (доступны только через ILRD)

| Metric | Описание | ID |
|--------|----------|----|
| **Active Users** | Уникальные пользователи по нашему трекеру | m37 |
| **Session Length** | Средняя длина сессии | s2 |
| **Sessions per User** | Сессий на пользователя | s5 |
| **Impressions per Session** | Показов на сессию | s6 |
| **Ad Session Length** | Длина рекламной сессии | s1 |
| **Revenue by SDK Version** | Доход с разбивкой по версии SDK | d1 |
| **Revenue by App Version** | Доход с разбивкой по версии приложения | d2 |

### Новые Filters

| Filter | Значения |
|--------|----------|
| **SDK Version** | Версия CAS SDK |
| **App Version** | Версия приложения |

### Новые Splits

| Split | Что делает |
|-------|-----------|
| **SDK Version** | Строка на каждую версию SDK |
| **App Version** | Строка на каждую версию приложения |

### Real-time: период "сегодня"

ILRD позволяет показывать данные за текущие сутки (gap между API и сейчас). Открытый вопрос: отдельный таб или продолжение графика с пометкой "expected".
