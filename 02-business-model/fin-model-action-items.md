## 2. ИЗМЕНЕНИЯ В РАСЧЕТАХ И ЛОГИКЕ МОДЕЛИ

### 2.1 Изменение периода учета новых клиентов
- **Было:** учитывались только клиенты, зашедшие в текущем месяце
- **Нужно:** брать клиентов за 2-3 месяца (включая предыдущий), так как клиент, пришедший в январе, никогда в январе денег не показывает

### 2.2 Добавление воронки лидов с конверсией
- Сумма DAO всех потенциальных клиентов = ~1.5 млн
- Конверсия = 20-25%
- Ожидаемый доход = 300-400 тыс.
- Цель: иметь prediction, сколько конвертируется из pipeline

### 2.3 Взвешивание лидов по категориям
- Разработать методику оценки лидов без данных от сторонних сервисов
- Классификация: L0, L1, L2 с соответствующими средними значениями (L1 ~ $6,000)

## 3. НОВЫЕ СТРОКИ/СЕКЦИИ В МОДЕЛИ

### 3.1 Force Majeure / Major Client Risk
- Учет сезонных просадок крупных VIP-клиентов (турецкие музыкальные приложения)
- Текущее значение: минус $180-200K к плану
- Компоненты:
  - Сезонное падение крупного турецкого клиента: -$180K
  - Digital Turbine просадка: -$20K

### 3.2 Отдельный учет Publishing
- **Было:** Publishing считался как пропорция от медиации
- **Нужно:**
  - Отдельная вкладка/секция для Publishing
  - Разделение Acquisition на медиацию ($120K) и publishing ($120K)
  - Учет старого publishing для VIP Monetization Setup

### 3.3 Новый BD (Вика) в модели
- Добавить второго BD менеджера с планом $240K (аналогично Игорю)
- Потенциал с синергетическим эффектом: $1300K от Вики к концу года
- Общий таргет двух BD: $480-600K новых клиентов

## 4. КОРРЕКТИРОВКИ ASSUMPTIONS (ДОПУЩЕНИЙ)

### 4.1 VIP Monetization Setup
- Плановый uplift: +15% для 25% топ-клиентов
- Фактический результат января: ~$48K из $60K (80% выполнения)
- **Открытый вопрос:** Можно ли охватить больше 25% клиентов? Будет ли uplift 15% или 8%?

### 4.2 R&D Optimal Setup
- Текущий статус: $10K пока игнорируется
- Причина: без BI невозможно оценить uplift "средний по больнице"
- Допущение в модели: к концу года новая версия SDK даст +8% буста трафика при раскатке на 80%

## 5. ИСПРАВЛЕНИЯ И УТОЧНЕНИЯ

### 5.1 Дискрепанси в отчетах (расхождение $10-23K)
- Проблема: цифры не совпадают между разными источниками
- Решение: стандартизировать формирование отчетов "по одному клику"
- Исключить: ошибки фильтрации ("бомжи-ферма", "промоутинг" и др.)

### 5.2 Исключение выбросов для анализа
- Правило: при расчете трендов исключать сезонных клиентов (турецкие музыкальные приложения)
- Пример: без двух турецких клиентов компания не просела, а выросла на $113K

### 5.3 Разграничение источников прироста
- Разделить прирост $113K на составляющие:
  - VIP Monetization Setup: ~$40-50K
  - Новые клиенты декабря: ~$50K
  - Возврат старого клиента: остаток

## 6. НОВЫЕ АНАЛИТИЧЕСКИЕ ЗАДАЧИ

### 6.1 Ретроспективный анализ Publishing за 2025 год
- Сколько клиентов пришло
- Сколько publishing смог обработать
- На сколько в среднем клиент вырос
- Цель: посчитать коэффициент конверсии и роста

### 6.2 Анализ источников новых клиентов
- Разделение лидов по виду услуги: медиация vs publishing
- Пример января: из 50 лидов — 20 на медиацию, 30 на publishing

### 6.3 Pipeline воронка для Publishing
- Входящий pipeline: 30 клиентов x $6K = $180K
- Минус churn (30%) = $60K остается
- Рост в 5x через publishing = $300K выход

## 7. ОРГАНИЗАЦИОННЫЕ УЛУЧШЕНИЯ

### 7.1 Единый источник истины для отчетов
- Проблема: разные данные у разных людей ($3800K vs $2000-2100K)
- Причина: разные периметры учета (CAS + PSV vs только CAS)
- Решение: четко определить, что входит в периметр модели

### 7.2 Автоматизация отчетности
- Текущий статус: без автоматизации, много ручной работы
- Блокер для R&D Optimal Setup: нет данных из Event Server (данные терялись 2 недели)
