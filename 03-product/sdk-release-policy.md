# CAS.AI SDK — Регламент релизов

> **Контекст:** результат встречи 25.02.2026 (Osyka, Shevnin, Kapatsyn, Yeshchenko) + ресерч релизных процессов AppLovin MAX, Google AdMob, ironSource/LevelPlay, Appodeal.
> **Статус:** Черновик v.0.1

## Зачем этот регламент

SDK — основной продукт CAS.AI. От его стабильности напрямую зависит выручка клиентов и наша. Сейчас процесс релизов имеет системные проблемы, которые этот регламент призван устранить.

### Вводные 

1. **Версии выходят без тестирования.** SDK попадает в публичный доступ сразу после сборки. Клиенты фактически являются тестировщиками. Были случаи ухода крупных клиентов со словами «SDK сырой».

2. **Нет понятия «стабильная версия».** Менеджеры не могут однозначно сказать клиенту, какую версию ставить. Попытка раскатить стейбл на топ-клиентов провалилась — оказалось технически и операционно невозможно.

3. **Бесконтрольное добавление фич.** Несколько человек одновременно заказывают фичи у SDK-разработчика. Фичи попадают в сборку без приоритизации, плана и приёмки. Заказчики не возвращаются за результатом месяцами.

4. **R&D без плана и метрик.** Эксперименты (bidfloors, конфигурации) ведутся без трекера, дедлайнов и оценки результата. Непонятно, какой статус, кто отвечает, когда закончим.

5. **Скрытая деградация перформанса.** Непротестированные фичи не крашат приложение, но снижают eCPM, ARPDAU, fill rate. Эффект виден только через месяцы на больших данных. К этому моменту уже поздно.

6. **QA тестирует в вакууме.** Тестовые конфигурации не воспроизводят реальные условия медиации (Max, GuaranteedSource, waterfall, geo-фильтры). Баги находятся только в продакшене у клиентов.

7. **Бета-версии уходят в пустоту.** Денис собирает бета, выкладывает — и месяцами ждёт фидбэка, которого нет, потому что нет назначенных людей и процесса.

## Треки версий

SDK поддерживает три параллельных трека. Каждый трек имеет свой уровень риска и аудиторию.

| Трек | Аудитория | Что входит | Пример версии |
|------|-----------|------------|---------------|
| **Alpha** | Только внутренние приложения (3–5 шт.) | Экспериментальные фичи из R&D-бэклога | 4.6.0-alpha |
| **Beta** | Группа лояльных клиентов + собственные приложения | Новые фичи, прошедшие Alpha | 4.6.0-beta |
| **Stable** | Все клиенты | Только обновления адаптеров и критические багфиксы | 4.5.x |

### Stable

- Содержит только проверенные компоненты.
- Обновления внутри трека — исключительно подъём версий рекламных адаптеров и исправление критических багов.
- Новые фичи в Stable **не добавляются** — они попадают сюда только после прохождения Alpha → Beta.
- Это версия, которую менеджеры рекомендуют клиентам по умолчанию.

### Alpha

- Внутренний трек для проверки новых фич.
- Интегрируется в 3–5 собственных приложений компании.
- QA проверяет: сборку под все платформы/фреймворки, отсутствие крашей, корректность показов, работу дебаг-режима.
- Никакие внешние клиенты Alpha-версию не получают.

### Beta

- Версия, прошедшая Alpha без критических проблем.
- Распространяется на группу лояльных клиентов (5–10 паблишеров), которые дали согласие на участие.
- Клиенты ставят Beta на часть своих приложений, не на весь портфель.

**Критерии отбора** (разнообразие профилей):
- Минимум 1 крупный паблишер (>100K DAU), 2–3 средних (10–100K DAU), 2–3 маленьких (<10K DAU).
- Обязательно: iOS + Android, Unity + native.
- Предпочтение: разные geo (CIS, LATAM, SEA) и рекламные конфигурации (waterfall, Max, bidding).

## Версионирование

### Core SDK

Формат: `MAJOR.MINOR.PATCH` (семантическое версионирование).

| Компонент | Когда меняется | Пример |
|-----------|---------------|--------|
| **MAJOR** | Ломающие изменения API, смена архитектуры | 4.x → 5.0.0 |
| **MINOR** | Новые фичи, прошедшие цикл Alpha → Beta → Stable | 4.5.x → 4.6.0 |
| **PATCH** | Подъём адаптеров, хотфиксы внутри Stable | 4.5.2 → 4.5.3 |

> Референс: AppLovin MAX, Google AdMob и Appodeal используют идентичную схему. ironSource/LevelPlay — аналогично. У всех четырёх major-версия меняется примерно раз в год.

### Адаптеры

Формат: `{версия_сети}.{патч_адаптера}`

Пример: адаптер для Meta SDK 6.21.0 → версия адаптера `6.21.0.0`. Если найден баг в адаптере (без изменения Meta SDK) → `6.21.0.1`.

Это делает совместимость самодокументируемой: по номеру версии адаптера сразу видно, на какую версию сети он рассчитан.

> Референс: AppLovin, AdMob и Appodeal (с 4.0.0) используют именно эту схему. ironSource использует свою нумерацию адаптеров (`5.x.0`), но документирует сертифицированную версию сети отдельно.

### Декаплинг адаптеров от Core SDK

Каждый адаптер объявляет **диапазон совместимых версий Core SDK**, а не привязку к конкретной версии. Это означает:

- Подъём Core SDK **не требует** обновления всех адаптеров.
- Подъём адаптера **не требует** обновления Core SDK (если текущая версия входит в диапазон).
- Адаптер помечается как «обновление обязательно» только если текущий Core SDK выходит за пределы поддерживаемого диапазона.

> Референс: ironSource внедрил декаплинг начиная с LevelPlay 8.4.0. Appodeal сделал аналогичный переход в 4.0.0 — убрал единый бандл `full` и перешёл на отдельные координаты `com.appodeal.ads.sdk.adapters.*`. До декаплинга каждое обновление SDK требовало обновления всех адаптеров — это был основной источник жалоб паблишеров.

## Критерии перехода между треками

### Alpha → Beta

| Критерий | Требование |
|----------|------------|
| Срок работы в Alpha | Не менее 2 недель |
| Крашрейт | Не выше baseline Stable-версии |
| QA-заключение | Все заявленные фичи протестированы, критических багов нет |
| Платформы | Проверено на iOS + Android, все поддерживаемые фреймворки (Unity, native) |
| Конфигурации | Проверены реальные (live) конфигурации медиации, не только тестовые (Max, GuaranteedSource, core waterfall) |
| Mediation Debugger | Все сети инициализируются, показы загружаются, логи чистые |

### Beta → Stable

| Критерий | Требование |
|----------|------------|
| Срок работы в Beta | Minor: не менее 1 месяца. Major: не менее 2 месяцев |
| Охват в Beta | Суммарно не менее 500K DAU на Beta-версии |
| Перформанс | A/B-сравнение подтвердило: ARPDAU не хуже Stable (см. «A/B тестирование SDK-версий») |
| Фидбэк клиентов | Нет открытых критических жалоб от бета-участников |
| Решение о переходе | Принимает Release Manager совместно с R&D Product Manager |

---
### Метрики

| Тип | Метрика | Порог |
|-----|---------|-------|
| **Primary** | ARPDAU | Не хуже Stable (≥ −2% при 95% confidence) |
| **Secondary** | eCPM, fill rate, impressions/DAU | Информативно, без жёсткого порога |
| **Guardrail** | Crash rate, ANR rate, retention D1/D7 | Не хуже Stable |


### Принятие решения

Release Manager и R&D Product Manager совместно оценивают результат:
- ARPDAU ≥ Stable и guardrails в норме → переход в Stable.
- ARPDAU ниже Stable, но в пределах −2% → продлить тест на 2 недели.
- ARPDAU ниже −2% или guardrails деградировали → версия не переходит в Stable. Анализ причин, возврат в Alpha.

> Референс: Appodeal документирует A/B медиаций с ARPDAU как primary metric и разделением через Firebase Remote Config. AppLovin рекомендует аналогичный подход при переходе между SDK-версиями.

## Deprecation Policy (политика устаревания)

### Статусы версий

| Статус | Определение | Реклама показывается? | Поддержка? |
|--------|-------------|----------------------|------------|
| **Supported** | Текущая + предыдущая major-версия (N, N-1) | Да | Да |
| **Deprecated** | Две major-версии назад (N-2) | Да | Нет |
| **Sunset** | Три и более major-версий назад (N-3+) | Может прекратиться | Нет |

### Сроки

- Major-версия выходит **раз в год** (фиксированная каденция, ориентировочно Q1). Это делает таймлайн deprecation предсказуемым для паблишеров.
- Версия остаётся **supported ~2 года** (пока не выйдет N+2).
- **Deprecated** — ещё ~1 год. Реклама показывается, но баги не фиксятся.
- **Sunset** — реклама может перестать показываться. Паблишер получает уведомление за **6 месяцев** до sunset.

> Альтернатива (не принята): привязка deprecation к minor-версиям (как у Appodeal: N, N-1 supported; N-2 deprecated; N-3 sunset по minor). Это даёт более короткий цикл, но вынуждает паблишеров обновляться каждые 2–4 месяца, что для нашей аудитории избыточно.

### Пример расчёта

| Версия | Статус | Выпуск | Deprecated с | Sunset с |
|--------|--------|--------|-------------|----------|
| 5.x | Supported | Q1 2027 | ~Q1 2029 | ~Q1 2030 |
| 4.x | Supported | сейчас | ~Q1 2028 | ~Q1 2029 |
| 3.x | Deprecated | — | сейчас | ~Q1 2027 |

> Референс: Google AdMob — единственный из крупных, кто публикует формальную deprecation policy с точными датами sunset для каждой major-версии. Appodeal публикует deprecation policy по minor-версиям (N/N-1 supported, N-2 deprecated, N-3 sunset) с конкретными датами. AppLovin и ironSource формальных сроков не публикуют, но де-факто следуют похожему циклу ~1–2 major-версии.

## Возможные Роли

### SDK Lead

- Разработка и сборка всех версий SDK.
- Подъём адаптеров для Stable-трека.
- Реализация фич из утверждённого плана для Alpha/Beta.
- Передача каждой сборки в QA перед любым внешним распространением.
- Ведение changelog для Core SDK и каждого адаптера.

### QA Engineer

- Тестирование каждой сборки: совместимость платформ, фреймворков, рекламных сетей.
- Тестирование на **реальных (live) конфигурациях**, не только тестовых — включая комбинации Max, GuaranteedSource, waterfall.
- Использование Integration Test Suite как первого шага проверки.
- Скачивание release-сборки из маркета, проверка логов в live-окружении.
- Ведение чек-листов и фиксация результатов.
- Формирование заключения: готова ли версия к переходу на следующий трек.

### R&D Product Manager (владелец R&D-бэклога)

- Приоритизация фич для следующей Alpha-версии.
- Сбор запросов от монетизационной команды, бизнеса, внутренних заказчиков.
- Формирование плана: какие фичи входят в следующий цикл, с оценкой ожидаемого эффекта.
- Отслеживание результатов: сработала ли фича по метрикам.
- Основная метрика роли — uplift выручки SDK-версий, выпущенных из R&D.

### Release Manager (возможно совместить с QA)

- Контроль процесса прохождения версии по трекам (Alpha → Beta → Stable).
- Право блокировать переход, если критерии не выполнены.
- Координация с менеджерами по раскатке Stable на клиентскую базу.
- Ведение календаря релизов.
- Публикация changelog и release notes.
- Отслеживание версий у клиентской базы: кто на какой версии, кто на deprecated/sunset.

> R&D Product Manager и Release Manager — это разные роли. R&D наполняет версию фичами. Release Manager гарантирует, что версия стабильна. Совмещение этих ролей в одном человеке нежелательно — возникает конфликт интересов.

## Процесс добавления фичи в SDK

```
Идея → R&D-бэклог → Приоритизация → Alpha-сборка → QA → Beta → Аналитика → Stable
```

### 1. Подача идеи

Любой сотрудник может предложить фичу. Заявка оформляется в трекере с описанием:
- Что делает фича.
- Какую бизнес-метрику улучшает и на сколько (гипотеза).
- Какие конфигурации/платформы затрагивает.

### 2. Приоритизация

R&D Product Manager собирает бэклог и раз в цикл (перед началом новой Alpha) определяет состав фич. Учитываются:
- Ожидаемый бизнес-эффект.
- Технический долг и запросы SDK Developer.
- Обновления рынка (новые рекламные сети, форматы).
- Ёмкость команды на цикл.

Решение фиксируется в трекере. Фичи, не вошедшие в цикл, остаются в бэклоге.

### 3. Разработка и Alpha

SDK Developer реализует утверждённый набор фич. После сборки — передача в QA. QA тестирует на внутренних приложениях с live-конфигурациями. Если критических проблем нет — версия получает статус Alpha.

### 4. Beta

Release Manager инициирует переход в Beta. Версия передаётся бета-группе клиентов. Параллельно аналитик отслеживает перформанс по BI: сравнение метрик Beta vs Stable на сопоставимых когортах.

### 5. Stable

При выполнении всех критериев Release Manager объявляет версию новым Stable. Публикуется changelog и migration guide (при смене major-версии). Менеджеры начинают раскатку на клиентскую базу.

## Migration Guide (при смене major-версии)

При выпуске новой major-версии (например, 4.x → 5.0.0) публикуется документ миграции:

| Раздел | Содержание |
|--------|-----------|
| Breaking changes | Список удалённых/изменённых API с заменами |
| Минимальные требования | Android API level, iOS version, Unity version, Kotlin version |
| Адаптеры | Какие адаптеры требуют обновления, какие совместимы |
| Шаги миграции | Пошаговая инструкция с примерами кода (старый → новый) |
| Известные проблемы | Текущие ограничения новой версии |

> Референс: Google AdMob публикует migration guide для каждого перехода major-версии (v22→v23, v23→v24, v24→v25) с точными таблицами замен API. ironSource для перехода 8.x→9.0.0 выпустил отдельный guide с тремя путями миграции (standard, UPM, смена формата).

## Правила для Stable-трека

1. В Stable попадают **только** версии, прошедшие полный цикл Alpha → Beta.
2. Внутри Stable допускается только подъём адаптеров и хотфиксы.
3. Хотфикс проходит ускоренный цикл в зависимости от severity (см. «Hotfix SLA» ниже). Без полного Beta-цикла, но с обязательным QA-заключением.
4. Менеджеры рекомендуют клиентам **только** текущую Stable-версию.
5. Если подъём адаптера вызывает регрессию — адаптер откатывается до предыдущей версии и помечается в changelog как «reverted».

### Hotfix SLA

| Severity | Описание | SLA на фикс | Процесс |
|----------|----------|-------------|---------|
| **P0** | Crashes, no fill, полная потеря монетизации | 24 часа | QA → Stable, без Beta |
| **P1** | Деградация eCPM/ARPDAU >10%, сломанный формат у части сетей | 3 рабочих дня | QA → Stable, без Beta |
| **P2** | Minor bugs, косметика, edge cases | Следующий Stable-патч | Стандартный цикл |

P0 и P1 хотфиксы выпускаются как patch-версия (e.g. 4.5.3 → 4.5.4). Release Manager уведомляет менеджеров о критическом обновлении для немедленной раскатки на клиентов.

> Референс: AppLovin практикует откат адаптеров — например, Facebook adapter 6.19.0.1 понизил Meta SDK с 6.19.0 до 6.18.0 из-за крашей баннеров, без обновления Core SDK.

## Ограничения

- Одновременно в Alpha может находиться **не более одной** версии. Нельзя параллельно разрабатывать несколько Alpha-веток с разными наборами фич. **Исключение:** P0/P1 hotfix-ветка для текущего Stable может существовать параллельно с Alpha.
- Заказчик фичи обязан участвовать в приёмке результата на этапе Alpha. Если заказчик не проверил фичу в течение 2 недель после сборки — фича снимается из версии.
- Фичи, не прошедшие через R&D Product Manager и трекер, **не включаются** в сборку.

## Сроки типового цикла

| Этап | Длительность |
|------|-------------|
| Приоритизация и планирование | 1 неделя |
| Разработка Alpha | 2–4 недели (зависит от объёма) |
| QA Alpha | 1–2 недели |
| Beta-период | Не менее 2 месяцев |
| Аналитика и решение | 1 неделя |
| **Итого минимум** | **~3.5–4 месяца от идеи до Stable** |

Цикл обновления адаптеров в Stable-треке — независимый, по мере выхода новых версий от рекламных сетей (ориентировочно 1–2 раза в месяц).

> Референс для сравнения каденций:
>
> | Платформа | Major | Minor | Адаптеры |
> |-----------|-------|-------|----------|
> | Google AdMob | 1 раз в год (Q1) | ~ежемесячно | Независимо от Core |
> | AppLovin MAX | ~1 раз в год | Каждые 4–8 недель | Независимо от Core |
> | ironSource LevelPlay | ~1 раз в год | Каждые 4–8 недель | Независимо (с 8.4.0) |
> | **CAS.AI (план)** | По необходимости | Каждые 3.5–4 мес. | Независимо от Core |

#### Механизм доставки Beta-версий

Beta публикуется в **отдельный репозиторий** (Maven / CocoaPods spec repo) с суффиксом `-beta`:
- Android: `com.cas.ai:sdk:4.6.0-beta.1` из `maven { url "https://repo.cas.ai/beta" }`
- iOS: `pod 'CASMediation', '4.6.0-beta.1'` из spec-репо `cas-beta`
- Unity: отдельный `.unitypackage` / UPM-пакет с beta-тегом

Паблишер переключается на beta-канал одной строкой в build-файле. Обратное переключение на stable — аналогично.

> Референс: Appodeal раздаёт бета через чекбокс в Appodeal Dependency Manager (Unity) — opt-in одним кликом. AppLovin публикует beta-версии в тот же Maven, но с `-beta` суффиксом.

## QA: Integration Test Suite

SDK может содержать встроенный инструмент самодиагностики, доступный паблишеру в дебаг-режиме.

### Что проверяет (минимум)

| Проверка | Описание |
|----------|----------|
| Версия SDK и адаптеров | Текущие установленные версии, наличие обновлений |
| Инициализация сетей | Статус каждой сети: успех / ошибка / таймаут |
| Загрузка рекламы | Тест-запрос по каждому формату (banner, interstitial, rewarded) для каждой сети |
| Конфигурация | Проверка app ID, ad unit IDs, наличие SKAdNetwork IDs (iOS) |
| Совместимость | Флаг, если адаптер устарел или Core SDK вне поддерживаемого диапазона |

### Активация

Включается вызовом `CAS.setDebugMode(true)` или специальным методом `CAS.showTestSuite()` перед инициализацией. В production-сборках автоматически отключён.

> Референс: AppLovin MAX — Mediation Debugger (runtime-проверка всех сетей, флаги версий, цветовая индикация). ironSource — LevelPlay Integration Test Suite (с 7.3.0, вкладки App Info / Networks / Ad Formats, тест загрузки рекламы по каждой сети). Appodeal — `startTestActivity()` (встроенная проверка адаптеров) + `setLogLevel()` для verbose-логирования waterfall.
