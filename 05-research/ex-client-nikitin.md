# Улучшения кабинета CAS (на основе casdev с Ильей Никитиным)

**Источник:** интервью interview-withex.md от 05.02.2026
**Респондент:** Илья Никитин (ex-клиент CAS, перешёл на AppLovin MAX)

---

## 1. Мультипользовательский доступ (критичная боль)

**Проблема:** Клиент вынужден делиться логином/паролем с коллегами (CMO, закупщики трафика, аналитики), нет возможности создавать пользователей.

**Улучшения:**
- Добавить управление пользователями в кабинете
- Реализовать систему ролей (владелец, аналитик, закупщик, только просмотр)
- Разграничить доступ к данным по ролям (не всем показывать всё)

---

## 2. API и экспорт данных для аналитики

**Проблема:** Клиенту нужны данные из рекламных кабинетов для сверки с MMP (AppsFlyer) и построения когортного анализа. Сейчас они используют кастомные ивенты SDK для отправки данных в Dev2Dev.

**Улучшения:**
- Предоставить Reporting API для выгрузки данных
- Revenue по сеткам, по дням, по кампаниям
- Формат: JSON/CSV для интеграции с внутренними BI-системами
- Документация по API

---

## 3. Интеграция с Dev2Dev / продуктовой аналитикой

**Проблема:** CAS не дружит с Dev2Dev — нет автоматической синхронизации данных медиации в продуктовую аналитику. Клиент вынужден пилить свои решения через кастомные ивенты SDK.

**Улучшения:**
- Интеграция "в один клик" с Dev2Dev (передача ad revenue)
- Альтернатива: партнёрство с Dev2Dev для нативной интеграции
- Как минимум — документация по настройке передачи ad revenue events

---

## 4. Дашборд "одной кнопкой"

**Проблема:** Клиент хочет зайти и сразу увидеть общую картину без настройки фильтров — "за январь рост 58%". Для маленьких команд это критично — экономит время.

**Улучшения:**
- Главный экран с ключевыми KPI: Revenue, рост за период, DAU
- Черновой PnL: Gross Revenue - примерная оценка
- Сравнение с прошлым периодом на главной

---

## 5. Документация (self-service)

**Проблема:** У AppLovin/ironSource есть раздел документации — можно разобраться самостоятельно без созвонов. У CAS — только через менеджера.

**Улучшения:**
- Раздел "Документация" в кабинете или отдельный docs-сайт
- SDK integration guides
- API documentation
- FAQ и troubleshooting
- Changelog обновлений

---

## 6. CMP (Consent Management Platform)

**Проблема:** Европейские консенты — головная боль. У AppLovin есть встроенный CMP "две кнопки нажал — заработало".

**Улучшения:**
- Встроенный CMP как часть SDK (опционально)
- Или готовый пошаговый гайд по настройке Google UMP
- Проверка compliance в кабинете

---

## 7. Когортный анализ с привязкой к UA

**Проблема:** Клиенту нужен анализ когорт по дням рекламных кампаний для прогнозных моделей. Ключевые точки: D7, D14, D30.

**Улучшения:**
- Когортные отчёты по ad revenue
- Привязка к дате установки (если доступна атрибуция)
- Экспорт когортных данных через API

---

## 8. Стабильность SDK

**Проблема:** От SDK "часто прилетали баги", приходилось писать обёртки для обработки ошибок, чтобы не падал ANR/crash rate.

**Улучшения:**
- Мониторинг качества SDK (crash-free rate)
- Changelog в кабинете с описанием фиксов
- Рекомендации по версиям SDK

---

## 9. Прозрачность по рекламным сетям

**Проблема:** CAS сам ведёт рекламные кабинеты — клиент не видит детали. Для Blended-кампаний в AppLovin нужна статистика рекламного трафика.

**Улучшения:**
- Детализация по рекламным сетям (SoV, eCPM по сетке)
- Видимость настроек waterfalls/bidding
- Отчёт по источникам трафика (для тех, кто закупается через CAS)

## Контекст респондента

- **Компания:** ~30 человек, из них маркетинг: CMO, закупщик, креативщики, 1 DE, 1 продуктовый аналитик
- **MAU:** ~70-80K (из них ~1.5M в США/Германии)
- **Модель:** гибрид IAP + реклама (~50/50)
- **Стек аналитики:** Dev2Dev + AppsFlyer + собственная BI на ClickHouse/AWS
- **Причина ухода:** необходимость закупки в AppLovin для Blended-кампаний
