## Что такое кабинет

Кабинет — основной продуктовый интерфейс CAS, через который клиент взаимодействует с платформой. 

Для клиента кабинет — это инструмент, чтобы понимать свой бизнес и управлять им в рамках CAS. Для команд CAS — инструмент управления, удержания и масштабирования.

---

## Цели кабинета

Кабинет обслуживает две стороны — клиента и бизнес CAS.

### Цели клиента (паблишера)

| Цель | Вопрос клиента | Что это в кабинете |
|------|---------------|-------------------|
| **Понять доход** | «Сколько я зарабатываю?» | KPI-карточки, график, таблица по приложениям |
| **Диагностировать проблему** | «Почему revenue упал?» | Drill-down по стране/сети/приложению, сравнение периодов |
| **Убедиться что всё работает** | «SDK подключён? Данные идут?» | Статус приложений, SDK version, integration health |
| **Получить деньги** | «Когда и сколько заплатят?» | Баланс, история платежей, invoices |
| **Управлять портфелем** | «Какие приложения, что с ними?» | Список приложений, статусы, в дальнейшем self-service |
| **Оптимизировать** | «Как заработать больше?» | Benchmarks, рекомендации, A/B тесты |
| **Управлять командой** | «Дать доступ, но не ко всему» | Роли, permissions, team management |
| **Не пропустить важное** | «Что сломалось? Что нового?» | Алерты, уведомления, SDK updates |

### Цели бизнеса CAS

| Цель | Зачем | Как кабинет это делает |
|------|-------|----------------------|
| **Удержание** | Клиент видит ценность → не уходит | Ежедневная привычка: зашёл → revenue → ушёл довольный |
| **Снижение нагрузки на support** | Меньше тикетов = дешевле | Self-service: клиент сам находит ответы |
| **Онбординг без менеджера** | Масштабирование без роста команды | Guided setup, checklist, integration status |
| **Upsell / cross-sell** | Mediation → publishing | «Ваш UA потенциал = X» — точка входа в publishing |
| **Доверие** | Прозрачность = меньше churn | Финансовая ясность, объяснение fraud adjustments |
| **Lock-in через привычку** | Больше пользуется → сложнее уйти | Saved reports, alerts, team setup, API |
| **Коммуникация** | Донести инфо без email/звонка | Баннеры, notifications, SDK update alerts |

### По стадии жизненного цикла клиента

| Стадия | Роль кабинета |
|--------|--------------|
| **Привлечение** | Демо-аргумент в продаже: «вот как выглядит ваш дашборд» |
| **Онбординг** | Чеклист: SDK → первый impression → первый отчёт |
| **Активация** | «Aha moment»: клиент видит свой revenue |
| **Ежедневное использование** | Привычка: утром открыл → KPI → закрыл |
| **Рост** | Upsell: «подключите publishing» |
| **Удержание** | Алерты, benchmarks, рекомендации — причины возвращаться |
| **Advocacy** | Клиент показывает дашборд коллегам → referral |

## Структура кабинета: 5 секций

Кабинет состоит из 5 секций. Каждая секция — группа связанных функций с общей целью.

### I. Аналитика — «сколько я зарабатываю и почему»

Центральная секция кабинета. Здесь клиент проводит 80% времени.

**a) Quick View** — обзор за 30 секунд. KPI-карточки (Revenue, DAU, ARPDAU, eCPM) с дельтами, график тренда, top-приложения. Ответ на вопрос «всё ли в порядке?». Это то, что клиент видит каждое утро.

**b) Reports** — глубокий анализ. Drill-down по приложениям/странам/сетям, сравнение периодов, экспорт, saved reports. Два контекста:
- **Mediation reports** — доходы от рекламы (Revenue, Impressions, eCPM, Fill Rate, по сетям)
- **Publishing reports** — UA-экономика (Installs, Spends, Ad Revs, InApp Revs, ROAS, RevShare)

OneBI — рабочее название аналитического блока. Спецификация: `03-product/design-spec-bi.md`.

### II. Приложения — «мой портфель и его здоровье»

Всё, что связано с конкретным приложением — в одном месте.

**a) Список приложений** — портфель с метриками (revenue, impressions, DAU за 7d), статусами (SDK, app-ads.txt, COPPA), кнопкой добавления. Поиск, фильтрация по статусу.

**b) Карточка приложения** — per-app dashboard: метрики, настройки, статус интеграции. Точка входа в drill-down по конкретному приложению.

**c) Монетизация** *(плейсхолдер для self-service)* — прозрачность: какие сети подключены, waterfall, floor prices. В будущем: клиент сам управляет waterfall, A/B тестирует конфигурации, включает-выключает сети.

Три операции, которые клиент должен уметь делать:

| Операция | Что конкретно | Кому нужно | Как сейчас |
|----------|--------------|------------|------------|
| **Управление приложениями** | Регистрация (iOS и Android — отдельно), bundle ID, название, ссылка на стор | Все | Частично в `/applications` |
| **Управление credentials сетей** | Ключи/аккаунты рекламных сетей клиента | Только L2+ (у L1 вшиты ключи CAS) | Вручную через саппорт |
| **Управление waterfall** | Порядок опроса сетей (от дорогой к дешёвой) + bid floors (минимальные пороги) | L2+ (тонкая настройка), L1 (дефолт) | Один JSON на всех, ручная кастомизация для крупных |

**d) Онбординг** — первое приложение: чеклист (SDK интегрирован → данные поступают → первый отчёт), прогресс-бар, контекстные подсказки. Чем быстрее клиент увидит данные — тем выше retention.

**e) SDK & API** — одна точка входа для всего, что связано с интеграцией: SDK keys, API keys, ссылки на документацию, test devices, downloads. Сейчас: только ссылка "Api documentation" в dropdown.


### III. Финансы — «когда и сколько мне заплатят»

Отдельная секция — потому что финансы это не аналитика и не настройки, а вопрос доверия. Если клиент не понимает свои деньги — он уйдёт.

**a) Обзор** — текущий баланс (available / pending), revenue за месяц, прогноз следующей выплаты. Ответ на «когда и сколько?» за 3 секунды.

**b) История транзакций** — все операции: выплаты, fraud adjustments, комиссии, компенсации. Фильтрация по периоду/типу, поиск, export CSV/PDF.

**c) Действия** — кнопка «заказать выплату» (когда баланс позволяет), настройка реквизитов, tax documents, payment threshold.

**d) Аналитика финансов** — monthly summary (сводка по месяцам), breakdown по типам операций. Прозрачность.

### IV. Аккаунт — «мой профиль и моя команда»

**a) Профиль** — персональные данные, API-ключи, OAuth-интеграции 3rd Party Tools: MMPs, UA & Ad Networks

**b) Команда** *(плейсхолдер)* — приглашение пользователей, роли (admin / viewer / finance), per-app доступ. Сейчас 1 аккаунт = 1 логин.

**c) Безопасность** — 2FA, forgot password.


### Developer Experience (DX) — сквозной слой

Клиент CAS — разработчик. DX не выделен в отдельную секцию, а проявляется внутри существующих:

| DX-задача | Где в кабинете | Что конкретно |
|-----------|---------------|---------------|
| Интегрировать SDK | **II. Приложения** → Онбординг | Code snippets, SDK key, пошаговый гайд (Android/iOS/Unity/Flutter) |
| Проверить что работает | **II. Приложения** → Карточка | SDK version, adapter versions, первый impression, test mode |
| Отладить проблему | **II. Приложения** → Карточка | Что вернула каждая сеть, latency, ошибки, логи |
| Обновить SDK | **V. Коммуникация** | "Вышла 4.6.0, ваша 4.5.2 — changelog + migration guide" |
| Подключить через API | **II. Приложения** → SDK & API | API keys, документация, rate limits, webhooks |
| Понять что сломалось | **I. Аналитика** + **V. Коммуникация** | Алерт "revenue упал после SDK update" + drill-down по SDK version |

### V. Коммуникация — «связь CAS с клиентом»

Всё, через что CAS говорит с клиентом за пределами отчётов.

**a) Технические уведомления** — алерты: падение revenue, проблемы с SDK, сбои сетей, обновления app-ads.txt. Три канала: in-app (bell icon), баннеры (критические), email (дайджесты). Ни один конкурент не делает нативные revenue-алерты — возможность для дифференциации.

**b) Upsell** — точки входа в дополнительные услуги CAS: «подключите publishing», «ваш потенциал UA = X». Контекстно: внутри аналитики, на карточке приложения, в monthly digest.

**c) Маркетинг и информирование** — анонсы новых фич, SDK updates, кейсы, best practices. Отдельно от технических алертов, чтобы не смешивать «надо починить» с «полезно знать».

## Принципы

### Managed service сейчас, self-service — цель

На текущем этапе кабинет не является self-service инструментом. Все ключевые операции (онбординг, настройка, изменения конфигураций) выполняются инженерами и командами CAS. Роль клиента — интегрировать SDK, опубликовать приложение и убедиться, что данные поступают.

Self-service — стратегическое направление. Путь: сначала прозрачность (показать клиенту что происходит) → потом контроль (дать управлять).

### Один интерфейс — разные данные

Кабинет адаптируется под тип клиента и показывает только тот объём информации и функциональности, который соответствует его роли, масштабу и модели взаимодействия с CAS.

## Кто использует кабинет

### Внешние клиенты
- **L1** — базовые клиенты медиации (подключил SDK, смотрит revenue)
- **L2** — продвинутые клиенты медиации (разбивки, сравнения, экспорт)
- **Pub** — публикуемые разработчики (UA + monetization)
- **PubC** — кандидаты в публикуемые (оценка потенциала)

### Внутренние роли CAS
- **Admin** — сводный отчёт по всему портфелю
- **Monetisation** — влияние SDK, A/B тесты
- **UA** — закупка трафика, ROAS
- **BD / Account Manager** — здоровье клиентов, upsell

Тип пользователя определяет scope данных, доступные метрики и дефолтные настройки. Интерфейс один — данные разные.

## Фаза 1: Рестайлинг (Q2 2026)

**Цель:** визуальный рестайлинг кабинета, чтобы OneBi (QuickView + Reports) встал нативно, а не чужеродным элементом. Функционал не меняем — только навигацию и оболочку.

**Источник:** встреча Alexei Shevnin + Serhii Shcherbyna, 2026-04-01 (`05-research/cabinet-tz-shcherbyna.md`)

### Навигация: 4 вкладки → 4 блока

| Было (4 вкладки) | Стало (4 блока) | Что внутри |
|-------------------|-----------------|------------|
| Mediation + Publishing | **Analytics** | QuickView (первый экран) + Reports (Mediation + Publishing в одном интерфейсе) |
| Applications | **Applications** | Список приложений + OAuth/credentials сетей (переезжают из Account) |
| Payments | **Payments** | Баланс (+ **баланс в хедере**, всегда на виду), история выплат |
| Account dropdown | **Profile** | Personal data, payment details, настройки |

### Новое в Analytics

- **QuickView** — первый экран после логина. KPI-карточки, Revenue Trend, Revenue by Network, выбор приложения/периода
- **Prediction Revenue** — виджет прогноза дохода, особенно для новых пользователей ("ваш потенциал = X")
- **Имена приложений из Store** — название + иконка платформы (английская локаль) вместо bundle ID
- **Reports** — Mediation + Publishing в одном интерфейсе с единым UX

### Новое в Applications

- **OAuth/credentials сетей** переезжают из Account dropdown (Authorize AdMob, DT Exchange) — плейсхолдер для будущего self-service подключения сетей

### Сквозные элементы

**Коммуникационная зона** (sidebar или все экраны):
- **Технические уведомления** — app-ads.txt, SDK updates, сбои сетей. Заменяют красные баннеры на приоритизированные нотификации
- **Маркетинговые коммуникации** — "подключите publishing", новости, блог, ссылки на Telegram/WhatsApp-группы
- **Human touch блок** — фото персонального менеджера + техсаппорт + аккаунт-менеджер с прямыми ссылками на WhatsApp/Telegram. Данные подтягиваются из 1С. Личный контакт — главное конкурентное преимущество CAS, но кабинет нигде его не показывает

**Онбординг тур** (первая сессия нового пользователя):
- Guided tour по интерфейсу (что где находится)
- Помощь с интеграцией SDK — "тут появятся ваши impressions"
- Friendly empty state вместо красных ошибок и прочерков — контент, обучение, prediction
- Зоны контакта с командой CAS (фото, ссылки, приглашение в чат)

### Проблемы для решения в рамках рестайлинга

| Проблема | Текущее поведение | Целевое поведение |
|----------|-------------------|-------------------|
| **Empty state** | Красная ошибка, прочерки, пустой график. Клиент уходит за 10 секунд | Friendly state: обучение, prediction, контакты команды |
| **app-ads.txt ложные тревоги** | Бот индексирует медленно или не индексирует. Красные кресты на 130+ приложениях → волки-волки | Убрать из первого экрана, приоритизировать, не показывать до подтверждения проблемы |
| **SDK race condition** | Клиент интегрирует SDK до регистрации приложения в CAS → алерты, некорректный config | Гейт или гайд: "сначала зарегистрируйте → потом интегрируйте" |
