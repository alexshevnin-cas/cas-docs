## Ключевые решения

| # | Решение | Контекст |
|---|---------|----------|
| 1 | **DDD (Domain-Driven Design)** — единый язык в коде, API и разговорах | Руслан предложил. Термины в коде должны совпадать с терминами Бориса и бизнеса |
| 2 | **UI и API — отдельные репозитории** | Решили с Вовой. Позволяет: заменить UI, сделать мобилку, дать API enterprise-клиентам |
| 3 | **Admin API — параллельный слой** | Не внутри основного API, а рядом. Управление менеджерами, кастомерами, бандлами |
| 4 | **Data Catalog нужен** | Каждое поле описано на техническом и бизнес-языке. Частично покрывает metrics-dictionary |
| 5 | **Real-time = раз в час/полчаса** | Не потоковый стриминг. Достаточно для клиентов |
| 6 | **Параллельный контур** | Новый кабинет живёт рядом со старым, не заменяет его сразу |

## Архитектура (согласованная на созвоне)

```
Sources → Data Lake → Normalization → Metrics → Views → API → Unified UI
                                                        → Admin API
```

- **Sources:** Ad Networks (API), CAS.SDK (ILRD), MMP, UA Networks, 1С
- **Data Lake → Views:** зона Бориса и дата-инженеров. Для Руслана = "таблички"
- **API:** отдельный слой, отдельный репозиторий. Контракты, документация
- **Admin API:** параллельный. Управление менеджерами, кастомерами, портфолио
- **Unified UI:** Quick View + Reports. Один UI для клиентов и мастер-юзеров (разные права)

## Инсайты

### Что подтвердилось
- Разделение Quick View / Reports — Руслан согласен, логично
- Metrics Layer как контракт между Борисом и Русланом — работает
- Итеративный подход (сначала B2B база, потом CH, потом ILRD) — принят

### Что нового
- **DDD как паттерн проектирования** — Руслан настаивает заложить сейчас, на начальном этапе. Термины в коде = термины в документации = термины Бориса
- **API для enterprise-клиентов** — Руслан сразу думает про внешний API с документацией ("как в банках — контракты, поля, запросы, ответы")
- **Аналитики хотят raw data** — Алексей: "будут люди, которые скажут — нахрен ваш UI, давайте мы скриптами в данные полезем". Это отдельный use case (Admin API + Raw Data)
- **Баг при миграции на CH** — поля в верхнем регистре вместо нижнего. Регистровая проблема сломала отчёт. Аргумент за Data Catalog
- **Процесс деплоя** — выкатили на прод не дождавшись фидбэка от клиентов. Нужен регламент: не катить без подтверждения

### Критерии успеха Этапов 1-2
1. Те же метрики, что сейчас (1:1 совпадение с текущим дашбордом)
2. Лучше эргономика и удобство (не эстетика — именно удобство взаимодействия)
3. Скорость работы интерфейса
